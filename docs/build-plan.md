# ClearGrant Analyzer Build Plan

## Build Philosophy

Build the complete, end-to-end extraction pipeline immediately. Use OpenAI's Structured Outputs to force the extracted document text into the Prisma JSON schema. Do not build mock dashboards; ensure the UI displays real data extracted from real files. Strict TDD and Git logging are required for the deterministic matching engine.

## Build Order

**Phase 1: Foundation & Data**
1. Initialize Next.js app with TypeScript and Tailwind.
2. Configure Supabase/PostgreSQL and Prisma.
3. Create Prisma schema (User, Profile, Grant, UploadedDocument, ExtractionResult, MatchResult) and run first migration.
4. Add authentication and protected layout.
5. Create global nav shell: Home, Upload Grants, Matrix Dashboard, Manage Profile, Logout.

**Phase 2: The User Profile**
6. Build the onboarding/profile questionnaire form.
7. Save structured user profile data to the database.

**Phase 3: The Engine (STRICT TDD REQUIRED)**
8. **EXECUTE HEADLESS TDD LOOP:** - You must build the deterministic matching logic (`utils/matchGrantToProfile`) using strict TDD.
   - Write the failing Vitest test -> run `npx vitest run > docs/tdd/logs/matcher-red.txt` -> write implementation -> run `npx vitest run > docs/tdd/logs/matcher-green.txt`.
   - **CRITICAL:** Immediately run `git add .` and `git commit -m "test(tdd): implement match logic with red/green logs"`.

**Phase 4: The Pipeline**
9. Build the "Upload Grants" page with drag-and-drop support for PDF, DOCX, and TXT.
10. Add server-side text extraction capabilities for those file types.
11. Build the LLM Extraction API Route: Send extracted text to OpenAI, enforce the JSON schema, and save the result to `ExtractionResult`.
12. Connect the pipeline: Upload -> Text Parse -> OpenAI JSON Extraction -> Database Save.

**Phase 5: The Interface**
13. Run the newly TDD'd matching engine to compare the extracted JSON against the User Profile, saving the `MatchResult`.
14. Build the Matrix Dashboard to display the analyzed, real database records.
15. Build the Grant Detail View to show the human-readable JSON breakdown.

## Rules

- Do not build CRM/application tracking.
- Do not add Submitted/Pending/Accepted/Rejected status.
- Do not build team features.
- Keep API keys server-side only.
- Keep each implementation task small.

## First Milestone Target

A logged-in user can:
1. Complete a profile.
2. Upload a real document.
3. Have the backend extract the text, query the LLM, and save the JSON.
4. See a matrix dashboard populated by their real data.
5. Click a grant detail page.
6. See match labels generated by the deterministic TDD logic.
